Networking Evidence

Concepts in Action

Here is how DNS, IP addressing, protocols, and security headers all play a role in making this site publicly accessible.

01 - DNS
Domain Resolution
nslookup / dig

DNS (Domain Name System) is basically the internet's phonebook. When you type a URL into your browser, a DNS resolver translates that human-readable name into an IP address so packets actually know where to go.

We ran nslookup against our GitHub Pages subdomain. The output showed the authoritative name server returned GitHub's edge IP addresses, confirming that DNS propagation completed successfully after we enabled Pages in the repo settings.

$ nslookup username.github.io
Server: 8.8.8.8
Address: 8.8.8.8#53
 
Non-authoritative answer:
Name: username.github.io
Address: 185.199.108.153
Address: 185.199.109.153
Address: 185.199.110.153
Address: 185.199.111.153

The four IP addresses returned are GitHub's CDN endpoints. The resolver used was Google's public DNS (8.8.8.8), and since it is not an authoritative server for this domain, the answer is marked as non-authoritative.

02 - IP
IP Addressing
IPv4 + IPv6

IP addresses are the actual delivery addresses on the internet. IPv4 uses 32-bit addresses (the classic dot-separated format), while IPv6 uses 128-bit addresses to handle the exhaustion of IPv4 space.

IPv4 Address
185.199.108.153
IPv6 Address
2606:50c0:8000::153
$ nslookup -type=AAAA username.github.io
Server: 8.8.8.8
Address: 8.8.8.8#53
 
Non-authoritative answer:
Name: username.github.io
Address: 2606:50c0:8000::153
Address: 2606:50c0:8001::153

Using the AAAA record query type returns IPv6 results. GitHub Pages supports both stacks, so clients that prefer IPv6 (like many modern mobile networks) can resolve and connect directly without fallback.

03 - Protocols
HTTP vs HTTPS
Headers + Status Codes

HTTP is the base protocol for transferring web content, but it sends data in plaintext. HTTPS wraps HTTP inside TLS (Transport Layer Security), encrypting everything in transit. GitHub Pages enforces HTTPS automatically.

$ curl -I https://username.github.io/cn-project1
 
HTTP/2 200
content-type: text/html; charset=utf-8
strict-transport-security: max-age=31556952
x-content-type-options: nosniff
x-frame-options: deny
x-xss-protection: 1; mode=block
cache-control: max-age=600
server: GitHub.com

The status code 200 means the request succeeded. The strict-transport-security header tells browsers to only ever connect via HTTPS for the next year (31,556,952 seconds). The x-frame-options: deny header prevents clickjacking by blocking the page from being embedded in iframes.

HTTP/2 (vs HTTP/1.1) also multiplexes requests over a single connection, which is why GitHub Pages defaults to it - it is noticeably faster.

04 - Security
HTTPS Enforcement
TLS + Security Headers

GitHub Pages has a built-in option to enforce HTTPS - any HTTP request gets automatically redirected to HTTPS with a 301 status. We enabled this in the repository settings under Pages.

The security headers served with every response add a few extra layers of protection:

strict-transport-security x-content-type-options: nosniff x-frame-options: deny x-xss-protection

HSTS (the strict-transport-security header) is especially useful - once a browser sees it, it will refuse to connect over plain HTTP at all. So even if someone typed "http://" manually, the browser would upgrade the request before it ever left the machine.

The TLS certificate is issued automatically by GitHub through Let's Encrypt, a free certificate authority. This is what puts the padlock in the browser and enables encrypted communication over port 443.
